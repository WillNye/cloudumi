    Configure Read Only Hub and Spoke Accounts¶

    You may configure Noq as read-only so that it does not have permissions to make changes within your AWS environment. You might want to use this configuration when Noq is deployed into highly sensitive environments or when running a trial of Noq prior to adopting.

    This is configurable per account, so some accounts can be read-only while others allow Noq to write changes. You will need to be logged in to Noq as an administrator to proceed with these steps.

    You may change a role from read-only to read-write by first deleting Noq's CloudFormation Stack on the target AWS account, followed by by removing the relevant role from Noq's Settings Page. You must remove both Spoke and Hub roles from your Settings page if you're changing the read-only state of your Hub Account.

    Procedure¶

    1. Navigate to Advanced->Settings¶

    Select Advanced > Settings from the Noq main menu.

    2. Optional: find the spoke role you'd like to change and remove it¶

    Under the Spoke Accounts section, identify the appropriate spoke account role.

    Once you find the correct account, Remove it.

    3. Add a New role¶

    Add a new role by clicking the New button on the Spoke Accounts section

    Login to AWS as an administrator for the targeted AWS account into which the read-only spoke account role should be installed.

    After clicking the New button, you will see a modal that prompts you to Add Spoke Account. Either click the Execute CloudFormation or the Copy URL button at the bottom.

    Note

    When you select the Copy URL button, a link is copied to your clipboard so that you may paste it into the desired browser session

    4. Execute the CloudFormation role template¶

    Before proceeding, please confirm that you are logged into the correct AWS account.

    The next screen that pops up is the CloudFormation Stack Create Stack screen.

    Select the ReadOnlyModeParameter boolean value and set it to either true (role is read-only) or false (role is read-write)

    Select I acknowledge that AWS CloudFormation might create IAM resources with custom names. and Create stack.

    After selecting Create Stack, your spoke role will be created into the selected account.

    5. Verify the read only spoke account¶

    Back on the Spoke Accounts page, click the browser's refresh button. You should now see the new account listed on this page.

    Noq will need to finish caching resources from the account. You may need to wait a few minutes before proceeding to the next step.

    Note

    In the Account Name field, there is a edit button on the right side of the text box (looks like a business card). You can use that button to rename the account.

    6. Test the read only spoke account¶

    Select Resources > Self-Service Permissions.

    Select a role or template to add permissions to and select Next

    Add a permission to an S3 Bucket (for instance), selecting the bucket under Bucket Name and hitting enter. Select the desired permissions. Be sure to select permissions that have a write effect (such as Put, Delete).

    After selecting Next you'll be required to enter a justification; notice that there should be a blue banner indicating that the policy request affects a read-only account. Click the Submit, Everything looks good button.

    You'll receive a link after submitting a request. Follow it to see your policy request. As a Noq administrator, you will be able to approve or reject the change. Approve the change associated with your read-only account.

    If desired, you may use temporary one-time credentials to write a change to your system.

    A modal should pop up prompting you to either provide temporary one-time credentials to allow Noq to write the change for you on the affected account, or to instruct Noq mark the request as approved without attempting to write the change. The latter option should be used if you've written the change outside of Noq.
